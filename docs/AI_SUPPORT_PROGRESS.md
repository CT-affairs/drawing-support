# AIサポート製図 PoC 進捗記録

最終更新: 2026-07-16

## 現在の到達点

Tfasから出力したDXFをブラウザからCloud Runへ送信し、`ezdxf`で読み込んでJSONレスポンスを返す一連の経路を確認した。

```text
Plesk上の /liff3/dxf-json.html
  ↓ multipart/form-data
Cloud Run drawing-support
  ↓ POST /api/v1/dxf/parse
ezdxfでDXFを読み込み
  ↓
構造化JSONをブラウザへ表示
```

## 実装済み

- `/liff3/drawing-support.html`を製図サポートのダッシュボード化
- `/liff3/dxf-json.html`をDXF JSON化の専用画面として追加
- 発注画面のサイドメニュー構成・配色・active表示を踏襲
- DXFファイルの選択、ドラッグ＆ドロップ、解析結果表示
- `POST /api/v1/dxf/parse`の実装
- `ezdxf`によるDXF基本情報のJSON化
- Cloud RunとPlesk間のCORS設定
- PoC中の接続確認のためCloud Runを公開設定へ変更
- フロントエンドFTP対象に`liff3/*.css`を追加
- 通常系・ファイル未指定・拡張子不正・破損DXF・サイズ超過のテスト
- レイアウト別のエンティティ件数、ブロック別の件数、INSERTとブロック名の取得
- DXFセクション、生レコード件数、ezdxf内部件数、監査件数などの診断情報
- 通常読み込みで図形が取得できない場合の`recover.read()`フォールバック
- 文字コード候補診断と選択結果の記録

## 実際に確認できたこと

約6,300KBのDXFを送信し、次のレスポンスを取得できた。

```json
{
  "dxf_version": "AC1009",
  "entities": [],
  "entity_counts": {},
  "layers": [],
  "schema_version": "1.0",
  "units": 6
}
```

これにより、以下は成立している。

- ブラウザからCloud Run APIへ接続できる
- 数MBのDXFを受信できる
- `ezdxf`でDXFとして読み込める
- DXFバージョンと単位を取得できる
- JSONレスポンスを返せる

## 現時点の課題

`entities`が空だったため、図面の図形情報をまだ取得できていない。

図形の全展開はまだ行わず、レイアウト別・ブロック別の集計とINSERT参照だけを追加した。Tfas出力DXFでは、図形が次の場所に存在する可能性がある。

- Paper Space
- Block定義
- INSERT参照先
- レイアウト別のエンティティ
- Tfas固有またはR12形式の要素

`AC1009`はAutoCAD R12形式であるため、次回はDXFの格納構造を診断する必要がある。ファイルサイズが大きいこと自体は、図形数が取得できない直接の原因とは限らない。

## 文字コード診断の確認結果

文字コードを確定してから通常解析または`recover.read()`へ進む二段階方式を追加した。実ファイルでは次の結果になった。

```json
{
  "dwg_codepage": null,
  "selected_encoding": "cp932",
  "selected_confidence": 0.85,
  "japanese_char_count": 83619,
  "replacement_count": 0
}
```

UTF-8およびCP1252は不正バイトで失敗し、CP932は置換なしで読み込めた。このため、対象DXFの文字コードはCP932である可能性が高い。一方、この判定を追加してもJSON全体の構造はほぼ変わらず、文字コードが図形取得不足の主原因ではないことが分かった。

レイヤー名・ブロック名には、なお文字化けまたはTfas固有の名称表現が残る可能性がある。これらの名称が読めるようになると、JSONの集計結果と実際の図面要素との対応を確認しやすくなる。

## 次回の作業候補

優先順に次を確認する。

1. レスポンスの`spaces`、`blocks`、`inserts`で実DXFの格納場所を確認する
2. `diagnostics.entities_section`の生レコード件数と種別を確認する
3. `diagnostics.encoding`の`dwg_codepage`、候補、選択確度を確認する
4. `diagnostics.loader`が`recover`になった場合の監査結果を確認する
5. CP932を前提に、レイヤー名とブロック名の元バイト列・読み込み経路を比較する
6. Tfasから出力するDXF設定（2D/3D、R12互換、ブロック化）を確認する
7. LINE、POLYLINE、TEXT、DIMENSION、INSERT等がどこに格納されているか確認する
8. 図形取得後に、AIへ渡す中間JSONの項目を整理する

### 今回進めた改修

`layers`、`entities[].layer`、`blocks[].name`、`inserts`内のレイヤー名・ブロック名を対象に、UTF-8のバイト列がCP932等の文字列として保存された文字化けを名称単位で復元する処理を追加した。厳密な逆変換が成功し、文字化け指標が明確に改善した場合だけ復元結果を採用する。正常な日本語名と英数字名は変更しない。

元の名称、復元名、変換方式、確度、出現回数、出現箇所は`diagnostics.name_decoding.mappings`へ残す。実ファイルで復元されない名称がある場合は、この診断情報と元DXFの該当名称を比較し、Tfas固有表現または別経路の文字化けとして候補判定を追加する。

実ファイルで確認した名称から、CP932の元バイトがCP1252文字とサロゲートエスケープへ分かれて残る経路も判明したため、逆変換を追加した。確認済みの例は`図面枠`、`通常`、`酸排気`である。この経路では復元前のバイト列を`source_bytes_hex`へ記録する。

## 名称復元の基本方針

DXFの文字化けは、出力元CAD、DXFバージョン、`$DWGCODEPAGE`、中間ツール、読み込みライブラリによって発生経路が異なる。今回判明した方式を全DXFへ固定適用するのではなく、複数の解析方法を「復元候補」として用意し、ファイルおよび名称ごとに妥当な候補を選択してJSON化する。

処理の基本フローは次のとおりとする。

1. DXF全体の文字コード、`$DWGCODEPAGE`、サロゲート、置換文字などを診断する
2. 元の名称を失わず、適用可能な復元方法から候補文字列を生成する
3. 厳密なバイト往復、未解決文字の有無、日本語文字数、文字化け指標の改善度を評価する
4. 同じレイヤー名・ブロック名・INSERT参照間で復元結果が一致するか確認する
5. 十分な確度がある候補だけをJSONの名称として採用する
6. 採用方式、確度、元名称、元バイト列、出現箇所を`diagnostics.name_decoding`へ記録する
7. 確度不足または候補なしの場合は元名称を保持し、将来の解析候補追加に備える

現在の候補方式は次のとおり。

| 候補方式 | 想定する文字化け経路 | 状態 |
|---|---|---|
| 元名称を維持 | 正常な名称、または安全に復元できない名称 | 実装済み |
| CP932等の文字列を元バイトへ戻してUTF-8化 | UTF-8のバイト列をCP932、CP1252、Latin-1等として保存・表示 | 実装済み |
| CP1252文字とサロゲートを元バイトへ戻してCP932化 | CP932のバイト列がCP1252表示され、未定義バイトが`\udcXX`として保持 | 実装済み・今回判明 |
| `$DWGCODEPAGE`を根拠にした直接復元 | DXFヘッダーと実際の名称バイトが整合するケース | 診断済み、選択精度を今後強化 |
| DXF Unicodeエスケープの復元 | `\U+XXXX`等が名称に未展開で残るケース | 未実装候補 |
| 出力元CAD固有ルール | Tfas等が独自の名称変換や識別子を使用するケース | 実ファイルごとに追加検証 |

今回確認した方式では、次の対応を再現できた。

| 復元前 | 元バイト列 | 復元後 | 用途の推定 |
|---|---|---|---|
| `*0-0\_001*\udc90}–Ê˜g` | `90 7d 96 ca 98 67`を含む | `*0-0\_001*図面枠` | 図面枠用レイヤー |
| `*0-1*’Ê\udc8fí` | `92 ca 8f ed`を含む | `*0-1*通常` | 通常要素用レイヤー |
| `*0-3*Ž_”r‹C` | `8e 5f 94 72 8b 43`を含む | `*0-3*酸排気` | 酸排気系統用レイヤー |

候補の自動選択で確度が拮抗する場合は、推測で確定せず、候補と診断根拠をJSONへ残して人が確認できる形へ拡張する。新しいDXFで別の文字化け経路が見つかった場合は、既存方式を置き換えるのではなく候補方式を追加し、再現用テストと確認例をこの文書へ追記する。

## セキュリティ・運用上の注意

現在のCloud Runは、Plesk上のブラウザから直接PoC APIを呼ぶため公開設定にしている。Cloud Runの公開設定はサービス単位であり、Cloud Run URL上のAPIを誰でも呼び出せる状態である。

一方、`/liff/`と`/liff2/`はPlesk側の別サービスであり、今回のCloud Run公開設定の対象ではない。

現在のPoCでは、利用者認証・認可は未実装である。機密図面や個人情報を含む本番データは使用しない。実運用前に、LIFF IDトークン等の認証検証、利用者・所属に基づく認可、監査ログ、ファイル保持方針を追加し、Cloud Run公開設定も見直す。

## 再開時の確認

- Cloud Runの最新リビジョンがデプロイ済みか確認する
- Plesk側の`/liff3/dxf-json.html`とCSSが最新か確認する
- Cloud Run URLが`liff3/dxf-json.html`の設定と一致しているか確認する
- 解析対象DXFは機密情報を除いたサンプルを使う
- まず診断用のModel Space／Paper Space／Block集計を追加する
