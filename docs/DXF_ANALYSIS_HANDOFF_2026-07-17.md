# DXF解析 引継ぎメモ（2026-07-17）

## 1. 目的

DXF解析の主目的は、図面から製作対象となるダクト候補を抽出し、次の処理へつなぐことである。

1. 在庫資材とのサイズ・材質等の突合
2. 製作対象の選定
3. 選定した対象の2D製作用図面への落とし込み

AIへ生のDXFや大量のLINEを直接解釈させるのではなく、Pythonとezdxfで座標・寸法・形状を正規化し、AIには候補の分類、注記との関連付け、判断理由の説明を担当させる方針とする。

## 2. 現在までに実装済みの範囲

- 基準環境をPython 3.12系、`ezdxf==1.4.4`へ統一
- DXF全体の文字コード候補解析
- レイヤー名・ブロック名の文字化け復元
- TEXT・MTEXT・DIMENSIONへの文字コード候補解析の適用
- JSON出力の厳密なUTF-8化
- Model Spaceの基本エンティティJSON化
- レイアウト別集計、ブロック別集計、INSERT一覧の出力
- `blocks[].entities`へのブロック内部構成要素の出力
- `blocks[].bbox`へのブロック定義座標での外接範囲出力
- ネストしたINSERTを含むbbox計算
- 空ブロックの`bbox: null`対応
- 文字復元方式、確度、出現箇所等の診断情報出力

`blocks[].bbox`は次の形式で、各配列の順序は`[X, Y, Z]`である。

```json
{
  "min": [0, 0, 0],
  "max": [20, 10, 0],
  "size": [20, 10, 0],
  "center": [10, 5, 0]
}
```

現在のbboxはINSERT配置前のブロック定義座標である。配置座標、回転、縮尺を反映したワールド座標bboxは未実装。

## 7. データ品質上の注意点

- `276.20999999998736`のような浮動小数誤差がある
- 一部の複雑なブロックに長さ0のLINEが含まれる
- JSON容量の大部分を重複した`blocks[].entities`が占める
- ワールド座標の位置・外接範囲はまだJSON化していない
- 現ファイルでは回転0、均一な正の縮尺だが、今後のDXFでは回転、非均一縮尺、負縮尺、MINSERTを考慮する必要がある
- JSONは厳密なUTF-8。未解決バイトの可視化22件は復元前文字列を保持する診断欄に限定され、実際のレイヤー名やTEXTは復元されている

座標を表示用に丸める場合も、元座標は失わない。形状比較では固定小数桁だけでなく、図面単位に応じた許容差を明示する。

## 8. 次回の最優先実装

### 8.1 INSERT変換後のワールド座標・bbox

各`inserts[]`へ、配置座標、回転、XYZ縮尺を反映した`world_bbox`を追加する。

```json
{
  "block": "-224--1170-BLOCK1171A",
  "world_bbox": {
    "min": [x, y, z],
    "max": [x, y, z],
    "size": [1000, 1150, 0],

  }

```

実装候補はezdxf 1.4.4の`INSERT.virtual_entities()`、`INSERT.matrix44()`、`ezdxf.bbox.extents()`。単純なローカルbboxのmin/max変換だけでは、回転時に誤るため、変換後形状またはbboxの全頂点を使って再計算する。

- 回転0、均一縮尺

## 12. INSERTと分類の標準設計（実装済み）

### 12.1 ブロック定義とINSERT配置の関係

Tfasから出力された正常なDXFも、ブロック定義とINSERT参照については標準DXFの構造として扱う。ブロック定義は同一DXF内で名前により一意に特定でき、INSERTはその名前を参照する。

```text
blocks[].name
  = 同一DXF内のブロック定義ID

inserts[].id
  = 図面上に配置されたINSERT個体のID

inserts[].block
  = blocks[].nameへの参照
```

1つのブロック定義を複数のINSERTが参照できるため、`block`名だけでは配置個体を識別しない。`inserts[].id`はINSERTごとに固有である必要があり、実装時は配列番号ではなくDXF entity handleを優先して使用する。handleを取得できない形式では、DXFファイル内で再現可能なフォールバックIDを検討する。

ブロック名の有効範囲は1つのDXFファイル内とする。複数DXF間で比較・参照する場合は、ファイルIDと組み合わせる。文字化け復元後の名前が衝突する可能性に備え、元の名前またはブロック定義handleを診断情報として保持する。

### 12.2 対象・メタ・判定不能の分類

`target`、`meta`、`unknown`の3値を、製作対象として扱う実体であるINSERT単位に付与する。`blocks[]`には同じ分類属性を重複して持たせず、形状定義・ローカルbbox・形状フィンガープリントを保持する。

```json
{
  "id": "insert:HANDLE123",
  "block": "BLOCK_A",
  "insert": [1000, 2000, 0],
  "rotation": 0,
  "scale": [1, 1, 1],
  "classification": {
    "role": "meta",
    "type": "title_frame",
    "confidence": 0.96,
    "evidence": [
      "large_sheet_like_bbox",
      "repeated_geometry",
      "title_frame_structure"
    ]
  }
}
```

分類値の意味は次のとおり。

| `classification.role` | 意味 | 後続処理 |
|---|---|---|
| `target` | 在庫突合・製作候補にする | `target_objects`の候補 |
| `meta` | 図面枠・表題欄・凡例等 | 図面範囲・尺度・注記の補助 |
| `unknown` | 自動判定できない | 人による確認対象 |

`classification.type`は`title_frame`、`rectangular_duct_candidate`等の詳細分類であり、後から拡張できる。自動判定の結果だけでなく、確度と根拠を必ず保持する。判定不能は失敗ではなく正式な状態として扱う。

### 12.3 判定の考え方

「ブロック単位で判定する」という表現は、ブロック定義の形状情報を材料にして、各INSERT配置を判定するという意味に整理する。最終的な分類の格納先は`inserts[]`に一本化する。

```text
blocks[]の形状・bbox・内部レイヤー・名称を評価
        ↓
各INSERTの配置座標・縮尺・周辺TEXTと組み合わせる
        ↓
inserts[].classificationへrole/type/confidence/evidenceを付与
```

レイヤーは分類結果ではなく、DXF原本の属性および判定材料として保持する。`_0-1_通常`のように、同じレイヤー内に対象形状と図面枠相当形状が混在するため、レイヤー単位で対象・メタを確定しない。

### 12.4 補足事項

- `inserts[]`は単なるメタ情報ではなく、ブロック定義を図面上へ配置した実体の一覧である
- `entities[]`内のINSERTと`inserts[]`は同じ配置情報を異なる用途で表す。後者は検索・解析用に整理した一覧である
- 同じブロックを複数位置へ配置した場合、各INSERTは別の対象候補として分類する
- ブロック内部に複数部品が含まれるケースは、当面は1 INSERT＝1候補で扱い、必要になった段階で`manufacturing_objects[]`へ分解する
- Model Spaceに直接存在するLINE、POLYLINE等は、INSERT分類とは別の第二段階で対象候補化する
- 元のDXF情報を削除せず、分類結果を追加情報として保持する

### 12.5 現在の実装範囲

以後のJSON化では`inserts[]`へ`id`、`handle`、`classification`を追加する。過去に作成したJSONは更新せず、アーカイブ済みデータとして扱う。

初版の自動分類は保守的なルールである。

- ブロック名・内部レイヤーに`図面枠`、`表題欄`、`title_frame`等があれば`meta`
- 内部レイヤーに`ダクト`、`排気`、`給気`、`換気`、`排煙`、`空調`等があれば`target`
- 単位がミリメートルで、十分に大きくシート状の矩形形状なら`meta`候補
- 根拠が不足する場合は`unknown`

INSERT自身のレイヤーは強い判定材料にしない。Tfas出力のように、INSERT全体が図面枠レイヤーへ置かれ、ブロック内部のレイヤーが実形状を表すケースがあるためである。分類結果は`diagnostics.object_classification`にも件数と戦略を記録する。
## Downstream process correction

The downstream manufacturing flow is `Tfas DXF -> drawing-support normalization -> JW-CAD 2D -> ALPHACAM CAM -> NC router -> field processing`. The JSON is an intermediate representation and must eventually be used to regenerate a cleaned DXF for JW-CAD. NCVC is not used in this project.
