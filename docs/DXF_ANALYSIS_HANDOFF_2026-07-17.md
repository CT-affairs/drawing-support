# DXF解析 引継ぎメモ（2026-07-17）

## 1. 目的

DXF解析の主目的は、図面から製作対象となるダクト候補を抽出し、次の処理へつなぐことである。

1. 在庫資材とのサイズ・材質等の突合
2. 製作対象の選定
3. 選定した対象の2D製作用図面への落とし込み

AIへ生のDXFや大量のLINEを直接解釈させるのではなく、Pythonとezdxfで座標・寸法・形状を正規化し、AIには候補の分類、注記との関連付け、判断理由の説明を担当させる方針とする。

## 2. 現在までに実装済みの範囲

- 基準環境をPython 3.12系、`ezdxf==1.4.4`へ統一
- DXF全体の文字コード候補解析
- レイヤー名・ブロック名の文字化け復元
- TEXT・MTEXT・DIMENSIONへの文字コード候補解析の適用
- JSON出力の厳密なUTF-8化
- Model Spaceの基本エンティティJSON化
- レイアウト別集計、ブロック別集計、INSERT一覧の出力
- `blocks[].entities`へのブロック内部構成要素の出力
- `blocks[].bbox`へのブロック定義座標での外接範囲出力
- ネストしたINSERTを含むbbox計算
- 空ブロックの`bbox: null`対応
- 文字復元方式、確度、出現箇所等の診断情報出力

`blocks[].bbox`は次の形式で、各配列の順序は`[X, Y, Z]`である。

```json
{
  "min": [0, 0, 0],
  "max": [20, 10, 0],
  "size": [20, 10, 0],
  "center": [10, 5, 0]
}
```

現在のbboxはINSERT配置前のブロック定義座標である。配置座標、回転、縮尺を反映したワールド座標bboxは未実装。

## 5. 形状重複の解析結果

414個のLINEのみで構成されたブロックを、bbox最小点を原点として平行移動差を除外し、端点を丸めて比較した。

| 項目 | 結果 |
|---|---:|
| LINEブロック | 414 |
| ユニーク形状 | 15 |
| 複数ブロックを含む同形状グループ | 10 |
| 重複形状グループに属するブロック | 409 |

主な繰返し形状は次のとおり。

- 配置後50 × 1250相当：104件
- 配置後50 × 800相当：87件
- 配置後1000 × 1150相当：81件
- 配置後1000 × 700相当：68件
- 配置後800 × 1150相当：20件
- 配置後800 × 700相当：17件

この結果から、ブロック名を個別の部材識別子として扱うよりも、形状フィンガープリントでプロトタイプを作り、各INSERTを配置インスタンスとして管理する構成が適している。

```text
形状プロトタイプ
  └─ 配置インスタンス
      ├─ ブロック名
      ├─ レイヤー
      ├─ 配置座標
      ├─ 回転角
      ├─ 縮尺
      ├─ ワールドbbox
      └─ 周辺TEXT
```

## 7. データ品質上の注意点

- `276.20999999998736`のような浮動小数誤差がある
- 一部の複雑なブロックに長さ0のLINEが含まれる
- JSON容量の大部分を重複した`blocks[].entities`が占める
- ワールド座標の位置・外接範囲はまだJSON化していない
- 現ファイルでは回転0、均一な正の縮尺だが、今後のDXFでは回転、非均一縮尺、負縮尺、MINSERTを考慮する必要がある
- JSONは厳密なUTF-8。未解決バイトの可視化22件は復元前文字列を保持する診断欄に限定され、実際のレイヤー名やTEXTは復元されている

座標を表示用に丸める場合も、元座標は失わない。形状比較では固定小数桁だけでなく、図面単位に応じた許容差を明示する。

## 8. 次回の最優先実装

### 8.1 INSERT変換後のワールド座標・bbox

各`inserts[]`へ、配置座標、回転、XYZ縮尺を反映した`world_bbox`を追加する。

```json
{
  "block": "-224--1170-BLOCK1171A",
  "world_bbox": {
    "min": [x, y, z],
    "max": [x, y, z],
    "size": [1000, 1150, 0],
    "center": [x, y, z]
  }
}
```

実装候補はezdxf 1.4.4の`INSERT.virtual_entities()`、`INSERT.matrix44()`、`ezdxf.bbox.extents()`。単純なローカルbboxのmin/max変換だけでは、回転時に誤るため、変換後形状またはbboxの全頂点を使って再計算する。

確認ケース：

- 回転0、均一縮尺
- 90度・任意角回転
- XYZ非均一縮尺
- 負縮尺による反転
- ネストしたINSERT
- MINSERT
- bboxを計算できないエンティティ

### 8.2 形状フィンガープリント

ワールドbboxと並行または直後に追加する。

- LINE端点の向きを正規化
- bbox原点へ平行移動して位置差を除外
- 設定した許容差で座標を量子化
- エンティティ順序を除外
- エンティティ種別、閉鎖状態、半径、角度等も含める
- 元データとフィンガープリント方式のバージョンを保持

想定フィールド：

```json
{
  "shape_fingerprint": {
    "algorithm": "local_geometry_v1",
    "value": "sha256:...",
    "tolerance": 0.001
  }
}
```

### 8.3 重複候補の診断

同じ形状フィンガープリント、ワールド座標、回転、縮尺を持つ配置をまとめる。ただし削除はせず、候補IDと根拠を返す。

### 8.4 TEXTとの空間的関連付け

ワールドbboxとTEXT挿入点の距離、包含関係、同一レイヤー、寸法表記パターンを使い、製作オブジェクト候補へ注記を付与する。

例：

- `1150×700`
- `フランジ`
- `消音機用`
- `FL+2500`

## 9. その後の処理順

1. `manufacturing_candidates`を生成
2. 幅、高さ、長さ、材質、板厚、系統、フランジ条件等を正規化
3. 在庫資材データとの突合API・データ契約を定義
4. 完全一致、加工可能、寸法不足、情報不足を分類
5. 製作対象を人が選択できる画面を用意
6. 選択対象から2D製作用データを生成
7. ezdxfでDXF出力し、TfasまたはJW-CADで確認

想定する中間データ：

```json
{
  "candidate_id": "duct-001",
  "system": "アンモニア排気",
  "object_type": "rectangular_duct_candidate",
  "dimensions": {
    "width": 1150,
    "height": 700,
    "length": null
  },
  "annotations": [
    "1150×700",
    "フランジフリー"
  ],
  "geometry": {
    "shape_fingerprint": "sha256:...",
    "world_bbox": {}
  },
  "stock_match": {
    "status": "unmatched",
    "candidates": []
  },
  "confidence": 0.0
}
```

`object_type`や寸法の意味付けは推論を含むため、根拠と確度を保持する。座標変換、bbox、距離、在庫寸法の適合判定はPython関数で再現可能にする。

## 11. 現時点の結論

文字化けしていたDXFから、意味のあるレイヤー、TEXT、ブロック内部形状、ローカルbboxを取得できた。さらに、INSERT縮尺を使った集計により、1000 × 1150、800 × 700等の実寸候補を機械的に確認できた。

これは「DXFを読める」段階から、「製作対象を寸法付きオブジェクト候補として抽出する」段階へ進んだことを意味する。次の中心課題はワールド座標化であり、その後に形状フィンガープリント、TEXT関連付け、在庫突合へ進む。
