# AIサポート体制でスピードアップを目指す製図・設計フロー

Tfas／DXF／AI／JW-CADを起点とした段階的な検証と、将来のIFC・BIM対応に向けた検討整理。

## 1. 目的

Tfasで共有された3D情報をもとに、社内ダクト加工用の図面をDXFで出力し、JW-CADで開いて現場作業に使える状態にする。

AIは、最終成果物を無条件に確定するものではなく、図面化・確認・修正を支援するものとして位置づける。

### 目指す状態

- 製図工程を10〜20%程度でも短縮する
- やり直しや手戻りを増やさない
- 1つの回答を断定するのではなく、複数の候補を出す
- 候補のうち少なくとも1つを、実務のたたき台として使える状態にする
- 将来、BIM利用が受注条件になっても移行しやすいデータ・システム構成にする

## 2. 主な業務フロー

```text
Tfas
  ↓ 3D情報の受領
図面化・加工用ルールの適用
  ↓
DXF出力
  ↓
DXF解析・候補生成（ezdxf）
  ↓
AIによる候補比較・チェック・修正案
  ↓
DXF再出力
  ↓
JW-CADで確認
  ↓
現場作業で使用
```

### 各工程

1. **情報受領**：Tfasで共有された3D情報を受領する
2. **図面化**：社内ダクト加工向けのルールに沿って図面化する
3. **DXF出力**：レイヤー、尺度、線種、座標、文字を確認する
4. **DXF解析**：図形・レイヤー・寸法・文字・ブロック等を構造化する
5. **AI支援**：複数候補の生成、比較、ルールチェック、修正案の提示を行う
6. **JW-CAD確認**：表示・編集・現場利用に問題がないか確認する
7. **フィードバック**：採用候補・修正内容・手戻りを次回検証へ反映する

## 3. 関係するツール・サービス

| ツール／サービス | 現状・役割 | 検討事項 |
|---|---|---|
| Tfas | 開発用アカウント作成済み。3D情報の起点 | データ取得方法、出力形式、属性情報、検証サンプル |
| ezdxf | DXFの解析・編集・再出力を担当する第一候補 | Tfas出力DXFとの互換性、対象エンティティ、文字・寸法・ブロックの保持 |
| JW-CAD | 無料で利用可能。現場作業用の確認・編集環境 | DXFの表示崩れ、尺度、線種、文字、編集性 |
| Cloud Run | ERP社内主幹システムの一部が稼働開始中 | 製図・AI支援用の専用プロジェクトを分離するか検討 |
| AI API | 候補生成・比較・チェック・修正案を担当 | 構造化出力、データ取り扱い、ログ、コスト、精度 |
| Rebro／BIM | 将来の受注条件になる可能性 | IFC入出力、属性保持、ツール差異を吸収する設計 |

## 4. DXF処理の第一候補：ezdxf

`ezdxf`はPythonのDXFライブラリで、DXFの読み込み、既存図面の変更、新規図面の作成、再保存に対応している。MITライセンスで、Python 3.10以降を対象としている。

- 公式ドキュメント：[ezdxf Documentation](https://ezdxf.readthedocs.io/en/stable/)
- 対応範囲：R12、R2000、R2004、R2007、R2010、R2013、R2018等
- ASCII DXF／Binary DXFの読み書きに対応
- レイヤー、ブロック、図形、文字、寸法等をプログラムから扱える
- DXFの内容をJSON化し、AIへ渡す前処理に利用できる

### ezdxfで最初に確認する対象

- `LINE`
- `ARC`
- `CIRCLE`
- `POLYLINE`
- `LWPOLYLINE`
- `SPLINE`
- `TEXT`／`MTEXT`
- `DIMENSION`
- `INSERT`（ブロック参照）
- レイヤー名、色、線種、線幅
- 図面単位、尺度、原点、座標系
- 拡張データ（XDATA）やTfas固有の属性があるか

### 制約

`ezdxf`はDXFファイルを扱うライブラリであり、高度なCADカーネルやBIMエンジンではない。複雑な3Dソリッド、特殊なプロキシオブジェクト、DWG変換、干渉判定等が必要になった場合は、ODA Drawings SDK等の商用SDKを比較する。

## 5. ezdxfを使った最初のPoC

### PoCの目的

Tfasから出力したDXFを読み込み、図面要素を正しく取得・構造化し、簡単なルールチェックとDXF再出力ができることを確認する。

### PoC手順

1. Tfasから代表的なDXFを1〜2種類出力する
2. JW-CADで開き、元図面の表示・編集状態を確認する
3. `ezdxf`でDXFを読み込む
4. レイヤー一覧とエンティティ種別を出力する
5. 各図形の座標、長さ、角度、外接範囲を取得する
6. 文字・寸法・ブロックの取得可否を確認する
7. 社内ルールに基づく簡単なチェックを行う
8. 修正または候補図面をDXFとして再出力する
9. JW-CADで再度開き、互換性を確認する

### 最小コード例

```python
import ezdxf

doc = ezdxf.readfile("input.dxf")
modelspace = doc.modelspace()

for entity in modelspace:
    print(entity.dxftype(), entity.dxf.layer)

doc.saveas("output.dxf")
```

### 最初に作る確認スクリプト

```text
scripts/
├─ inspect_dxf.py        # レイヤー・図形種別・座標範囲を出力
├─ extract_dxf_json.py   # AIに渡す構造化JSONを生成
├─ validate_dxf.py       # 社内ルールのチェック
└─ generate_candidate.py # 候補パターンをDXFとして出力
```

## 6. AIに渡すデータの考え方

AIにDXFファイルそのものを直接読ませるのではなく、`ezdxf`で図面情報を構造化してからAIへ渡す。

```text
DXF
  ↓ ezdxf
構造化JSON
  ├─ entities
  ├─ layers
  ├─ blocks
  ├─ dimensions
  ├─ texts
  ├─ bounding_box
  └─ validation_results
  ↓
AI API
  ↓
候補パラメータ／修正指示／チェック結果
  ↓
ezdxf
  ↓
候補DXF
```

### AIの出力は構造化する

AIからは、自由文ではなく、次のような構造化された出力を受け取る。

```json
{
  "candidates": [
    {
      "id": "candidate-01",
      "description": "標準曲げパターン",
      "changes": [],
      "warnings": [],
      "confidence": 0.0
    }
  ],
  "validation": {
    "passed": false,
    "issues": []
  }
}
```

AIには座標計算や寸法計算を完全に任せず、計算・図面生成はプログラム側で行う。AIは候補選択、ルール適用方針、異常箇所の説明、修正案の提示を担当させる。

## 7. DXFライブラリ／API候補の比較

| 候補 | 主な用途 | 採用判断 |
|---|---|---|
| **ezdxf** | PythonによるDXF解析・編集・生成 | PoCの第一候補。まずこれでTfas出力を確認する |
| **ODA Drawings SDK** | 高機能なCADデータ読み書き・編集 | ezdxfで保持できない要素がある場合に評価する |
| **Aspose.CAD** | REST API／.NET／PythonによるCAD変換・処理 | Cloud RunからAPIとして呼び出したい場合に比較する |
| **dxf-parser** | JavaScript／Node.jsでDXF解析 | Web画面やNode.js中心の構成で検討する |
| **dxf-writer** | JavaScriptによるDXF生成 | Node.jsで単純な候補図面を生成する場合に検討する |
| **netDxf** | C#によるDXF読み書き | .NET既存資産がある場合の候補。ただし更新状況に注意 |
| **CADLens API／MCP** | DXFを構造化JSONとしてAIから参照 | AIによる読み取り・検索の補助候補。実データで要検証 |

## 8. 目標の確認

短期の工程改善と、中長期のBIM移行準備を分けて評価する。

### 短期：工程改善

- 作業時間
- AIを使った場合の修正時間
- 候補の採用率
- 手戻り件数
- 担当者ごとの作業時間差
- JW-CADでの修正量

### 中長期：BIM移行準備

- IFCへの変換可能性
- 形状・寸法・接続・部材属性の保持率
- Rebro等の別BIMツールとの連携
- データ形式変更時の再開発量

## 9. Cloud Runの構成案

ERP社内主幹システムとは別に、製図・AI支援用の専用プロジェクトを設ける案を基本候補とする。利用者の入口・認証は共通化し、実行環境、サービスアカウント、データアクセス、AI APIの課金・上限は分離する方向が有力。

```text
共通入口・認証
        ↓
ERPプロジェクト       製図・AI支援プロジェクト
                      ├─ DXF解析サービス
                      ├─ ルールチェックサービス
                      ├─ AI連携サービス
                      └─ DXF生成サービス
```

## 10. 検証前に決める項目

- [ ] 対象図面・部材・形状を1〜2種類に限定する
- [ ] 現行作業の基準時間を計測する
- [ ] 許容する修正回数・手戻りを定義する
- [ ] 採用可能な候補の品質基準を決める
- [ ] DXFのレイヤー・尺度・線種・命名規則を整理する
- [ ] AIに渡すデータ項目と、渡してはいけない情報を整理する
- [ ] 最終確認者と現場展開者を決める
- [ ] Tfas出力DXFに特殊エンティティ・拡張属性がないか確認する

## 11. 当面のアクション

1. `ezdxf`の開発環境を作る
2. Tfasから代表DXFを取得する
3. `inspect_dxf.py`を作成する
4. レイヤー、図形種別、座標範囲、文字、寸法、ブロックを一覧化する
5. `extract_dxf_json.py`でAI入力形式を定義する
6. ルールチェックを1〜2項目だけ実装する
7. `generate_candidate.py`で単純な候補図面を出力する
8. JW-CADで開いて互換性を確認する
9. 作業時間と手戻りを現行手順と比較する
10. 対応できないDXF要素があれば、ODA Drawings SDK等を比較する

## 12. 将来フロー

```text
Tfas → IFC → BIM／Rebro → 構造化データ → AI → DXF → JW-CAD
```

将来的には、ツール名に依存せず、形状・寸法・接続・部材属性・変換ルールを中間データとして管理する。これにより、TfasからRebro等へツールが変わっても、AI支援・DXF生成・JW-CAD連携の部分を再利用しやすくする。

## 13. まとめ

まずは、Tfas→DXF→JW-CADの既存フローに`ezdxf`を追加し、DXFを正しく読み取り、構造化し、簡単な候補図面を再出力できるかを確認する。

AIは、図面を直接生成する唯一の判断者ではなく、複数候補の生成・比較・ルールチェック・修正案の提示を担当する。最終的な図面生成と寸法計算は、検証可能なプログラム処理として実装する。
