# AGENTS.md

このファイルは、AIサポートを含む開発作業で常に守る規約です。

## 作業開始時

- `README.md`、`REPOSITORY.md`、`docs/INTEGRATIONS.md`を読む。
- `git status`を確認し、利用者の未コミット変更を上書きしない。
- 未確定仕様は推測で固定せず、既存コード・文書を確認するか質問する。

## リポジトリ境界

- 製図支援の画面・API・データモデル・テスト・デプロイ設定はこのリポジトリで管理する。
- 日報ツールや帳票ツールのコードを、このリポジトリへ複製して独自改変しない。
- 他ツールとの連携は、URL遷移または明文化されたAPI・データ契約に限定する。
- 共有データを変更するときは、利用する全リポジトリへの影響を確認し、`docs/INTEGRATIONS.md`を更新する。

## 実装

- 秘密値をコード、テストデータ、文書、ログへ書かない。
- 認証・認可は別物として扱い、画面を非表示にするだけでアクセス制御を済ませない。
- 外部入力、ファイル、API応答は検証してから利用する。
- 変更には、通常系と主要な失敗系を確認できるテストまたは再現手順を添える。
- 公開ファイルを追加した場合は、デプロイ対象リストやCloud Build設定も確認する。

## ローカルPython環境

- Cloud Runバックエンド（`/liff/`日報ツール、`/liff2/`帳票・発注管理ツール、`/liff3/`本ツール）は、いずれもPython 3.12系・本リポジトリの`ezdxf==1.4.4`相当の基準環境で作業する。各リポジトリ直下に作成した`.venv`（`README.md`のセットアップ手順を参照）の中で実行・検証し、システム全体のPythonをこの基準へ変更しない。
- このPCではPython 3.8を別のオンプレサーバーの開発・保守用に維持する必要があるため、システムの既定Pythonをアンインストール・アップグレードしない。`.venv`を有効化しない素の`python`コマンドは古い環境を指す可能性があるため、作業前に`.venv`を有効化したか確認する。
- 日報ツール・帳票ツールのリポジトリで作業する場合も、同じ考え方（システムPythonではなくリポジトリごとの仮想環境を使う）を踏襲する。

## Git

- 意図したファイルだけをステージする。原則として無条件の`git add .`は使わない。
- コミット前に差分とテスト結果を確認する。
- pushやデプロイは利用者の明示的な依頼または合意後に行う。

## 文書の更新

次の変更ではコードと同時に文書を更新する。

- 公開URL、API、認証方式
- Firestore等のコレクション・フィールド
- 他リポジトリが利用する識別子
- デプロイ方法、環境変数
- リポジトリの責務境界
